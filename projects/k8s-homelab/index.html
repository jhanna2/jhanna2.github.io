Kubernetes Security Homelab

A security-first k3s cluster built from scratch on bare-metal Proxmox — starting deliberately broken, hardened systematically, with every decision documented. Built to close a hands-on Kubernetes security gap and produce a public artifact demonstrating how I approach security engineering when there's no ticket, no deadline, and no one telling me what to build.

View Source ↗

The Problem

Kubernetes security experience is easy to claim and hard to verify. Coursework covers the concepts. Cloud-managed clusters abstract the controls. What's missing is evidence of someone who actually configured RBAC from scratch, wrote admission policies, built detection rules, and documented why each decision was made — not just what was done.

Approach

The project runs in three phases, visible in git history via annotated tags.

  • Phase 1 — Deliberate misconfigurations. The cluster is deployed in an intentionally broken state: containers running as root, privileged flags set, wildcard RBAC bindings, secrets in plaintext environment variables, no network policies, hostPath mounts. Every misconfiguration is documented with its exploit path, blast radius, and MITRE ATT&CK mapping. The tag v0.1-insecure captures this state permanently.
  • Phase 2 — Systematic hardening. Every Phase 1 issue is fixed in a separate commit with a message explaining the security rationale. Controls implemented: RBAC, pod security standards, pod hardening, network policies, resource limits, secrets management via Vault injection, and Kyverno admission policies.
  • Phase 3 — Runtime detection. Falco deployed with custom rules targeting realistic attack scenarios. All alerts flow to OpenSearch. Simulation scripts validate each rule fires and the alert completes the round-trip to the dashboard.

Key Design Decisions

  • Start broken on purpose. Hardening a cluster you built correctly from the start proves you can follow a checklist. Starting from a documented attack surface and fixing it systematically proves you understand the threat model. The insecure tag in git is the evidence — not a claim.
  • One commit per control. The git log is the audit trail. Every RBAC change, every network policy, every Kyverno policy has its own commit with a message explaining the security rationale.
  • Raw YAML for own workloads, Helm for third-party tooling. Helm earns its place for Falco, OpenSearch, and Kyverno. For workloads you own, explicit manifests are the right choice.
  • LXC vs VMs for GPU-adjacent workloads. When full VM device passthrough is impractical, privileged LXCs with explicit device mapping can be a deliberate tradeoff at homelab scale — bounded by threat modeling and acceptance testing, not convenience alone.
  • Document every course correction. Honest pivots documented in writing are more credible than a history that only shows things that went right.

Direction

The hardened cluster is the foundation for the rest of the homelab stack — Falco detection engineering, a CTI agent pipeline, and an LLM safety platform built on top. Each layer is a separate project with its own documentation and git history, all rooted in this cluster.
