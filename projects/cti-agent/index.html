CTI Agent

A Go-based cyber threat intelligence pipeline running on a schedule inside a Kubernetes cluster. Pulls external threat bulletins, enriches IOCs in parallel against free threat intelligence feeds, triages findings against a live cluster asset inventory, uses a locally-hosted LLM for context-aware analysis, and generates OpenSearch detection rules written back to the SIEM.

View Source ↗

The Problem

Threat intelligence pipelines at production scale involve commercial platforms, paid feeds, and cloud infrastructure. Building one from scratch in a homelab — with free feeds, local inference, and self-hosted storage — requires making every architectural decision explicitly rather than accepting what a platform provides.

The problem this solves is also concrete: the CTI agent runs against the same OpenSearch instance that holds the cluster's asset inventory. When it processes a threat bulletin, it knows which cluster workloads are running the affected software and what their blast radius looks like. Triage isn't generic — it's against a specific, known environment.

Approach

External feeds (OTX, ThreatFox, URLhaus)
   └── Fetch + normalize bulletins
         └── Parallel IOC enrichment (goroutines per source)
               └── CMDB query — match affected assets
                     └── LLM triage (local model via Ollama / LiteLLM)
                           ├── Fact Card generation
                           ├── Risk assessment against CMDB
                           └── Executive summary
                                 └── Detection rules → OpenSearch monitors
                                       └── Enriched report → cti-reports index

Architecture

The agent is structured as a Go module with clear internal package boundaries. Enrichment runs concurrently — each source is called in a separate goroutine, results collected via channels. Total enrichment time is bounded by the slowest source, not their sum.

LLM calls route through the LiteLLM proxy in the llm-gateway namespace. The agent doesn't call Ollama directly — all traffic passes through the content sanitizer, TOON normalizer, and inline guards before reaching inference.

Key Design Decisions

  • Go over Python for the pipeline. The enrichment step benefits from real concurrency — goroutines and channels are the right tool. A single binary deployment is simpler to containerize than a Python service with dependency management.
  • Local inference over cloud APIs. A quantized open-weights model on consumer-class GPU hardware delivers enough throughput for scheduled triage without API keys or vendor-hosted prompts leaving the network. KEDA scale-to-zero keeps idle GPU memory from pinning the rest of the cluster.
  • CMDB-aware triage. The difference between generic threat intel and useful threat intel is context. When the CTI agent triages a bulletin, it queries the live asset inventory to understand whether affected software is running anywhere — and if so, what its blast radius looks like.
  • Fact Cards as the structured triage unit. Each processed bulletin produces a Fact Card — a normalized JSON schema capturing threat actor, TTPs, affected systems, IOCs, confidence scores, and CMDB matches. This keeps the LLM prompt focused and the output parseable.
  • Detection rules as first-class output. The agent doesn't just produce reports — it generates OpenSearch monitor configurations from triage findings and writes them back to the SIEM.

KEDA Scale-to-Zero

GPU inference is resource-intensive if left running continuously between sparse batch jobs. KEDA scales the inference deployment to zero when idle; a CronJob warms the service before each pipeline run so the first LLM call is not cold-start chaos, then autoscaling releases accelerator memory for other workloads.

Direction

The current implementation uses three free enrichment sources. The architecture is designed to add sources by implementing the enricher interface — paid feeds like Recorded Future or ThreatStream would drop in without changing the pipeline logic. The CMDB-aware triage is also worth extending: richer matching on network exposure, RBAC blast radius, and historical incident context.
